# PRD: «Встроенный тренер» — трекер силовых тренировок с движком прогрессии

**Версия:** 1.0 · **Дата:** 10.07.2026 · **Статус:** Draft

---

## 1. Purpose

Персональное веб-приложение, объединяющее журнал силовых тренировок и алгоритмического тренера. Продукт решает три проблемы, которые не закрывают ни бумажный блокнот, ни массовые трекеры (Hevy, Strong):

1. **Нет ответа «какой вес ставить сегодня»** — приложение рассчитывает рекомендацию на каждый подход по гибридной модели (двойная прогрессия + RIR-коррекция) и объясняет её логику.
2. **Прогресс не виден** — аналитика по e1RM, недельному объёму на мышечную группу и автоматический детектор стагнации.
3. **Нет управления усталостью** — календарные мезоциклы (5 недель роста → делоуд) встроены в план, а не оставлены на волю пользователя.

Целевой результат: устойчивый рост мышечной массы при контролируемом восстановлении, с трением ввода данных ≤ 5 секунд на подход.

## 2. Target Users

**Персона 1 — «Осознанный атлет 40+» (основная).**
Сергей, 46, руководитель. Тренируется 6 раз в неделю (3 силовые + 3 бега), цель — гипертрофия при сохранении здоровья суставов и восстановления. Фокус на longevity: доверяет данным, а не ощущениям. Хочет понимать *почему* алгоритм рекомендует именно этот вес. Пользуется телефоном в зале, ценит скорость ввода. Боль: прогресс встал, нет системы прогрессии и делоудов.

**Персона 2 — «Возвращающийся после перерыва».**
35–50 лет, тренировался раньше, вернулся в зал после 1–3 лет паузы. Помнит старые рабочие веса и рвётся к ним слишком быстро. Нуждается в консервативной рампе нагрузки и защите от перегруза (RIR-сигнал «слишком тяжело» → удержание веса). Боль: цикл «рванул — травмировался — бросил».

**Персона 3 — «Систематизатор без тренера».**
30–45 лет, тренируется стабильно 2–3 года, персонального тренера не имеет (дорого/не доверяет). Читает Israetel/Helms, знает термины RIR и мезоцикл, но не хочет вести Excel вручную. Боль: знает теорию, не имеет инструмента её применения.

## 3. User Scenarios

**S1. Тренировка дня (ядро продукта).** Понедельник, зал. Пользователь открывает приложение → выбирает день A → видит первое упражнение с рекомендацией: «Жим лёжа: 42.5 кг × 10, цель RIR 2» и прошлым результатом рядом. Выполняет подход, вводит вес/повторы/RIR за 3 касания → автоматически стартует таймер отдыха 2:30 → по сигналу переходит к следующему подходу. Проходит всю тренировку за 60 минут, весь ввод занял < 2 минут суммарно.

**S2. Реакция на прогресс.** Пользователь три сессии подряд достигает верха диапазона повторов (12) во всех сетах жима. Приложение на следующей тренировке рекомендует +2.5 кг с возвратом к 8 повторам и показывает причину: «достигнут потолок диапазона». e1RM на графике растёт.

**S3. Защита от перегруза.** После недели плохого сна пользователь вводит RIR 0 (отказ) на двух подходах при целевом RIR 2. Приложение не наращивает вес и рекомендует удержание с пометкой «фактическое усилие выше целевого — закрепляем вес». Через 5 недель цикла приложение объявляет делоуд-неделю: 60% веса, половина сетов.

**S4. Разбор динамики дома.** Вечером за чаем пользователь открывает экран аналитики: e1RM тяги растёт, жима — плато 3 недели (бейдж стагнации с подсказкой «сменить вариацию или проверить объём»). Диаграмма недельного объёма показывает: грудь 8 сетов/нед — ниже коридора 10–20. Пользователь добавляет упражнение на грудь в день C через конструктор.

**S5. Калибровка нового упражнения.** Детектор стагнации предложил заменить жим лёжа на жим гантелей — истории по нему нет. На первой сессии приложение включает режим калибровки: пользователь делает разведочный подход (16 кг × 15, RIR 4), приложение по e1RM проецирует рабочий вес под цель «10 повторов, RIR 2» → рекомендует 20 кг. Контрольный подход подтверждает прогноз (10 × RIR 2) — вес зафиксирован как база для прогрессии. Калибровка заняла 2 подхода вместо недели гаданий.

**S6. Перенос и сохранность данных.** Пользователь купил новый телефон. Скачивает JSON-бэкап на старом устройстве, открывает приложение на новом, импортирует файл — вся история, программа и позиция в мезоцикле восстановлены.

## 4. Success Metrics

| Метрика | Цель | Как измеряем |
|---|---|---|
| Полнота логирования | ≥ 90% плановых тренировок залогированы | сессий в логе / плановых сессий за 4 нед |
| Трение ввода | ≤ 5 сек на подход, ≤ 3 касания | замер по факту использования |
| Рост e1RM | +5–10% на ключевых базовых за мезоцикл (первые 2–3 цикла) | график e1RM |
| Объём в коридоре | все целевые группы 10–20 сетов/нед | диаграмма объёма |
| Соблюдение делоудов | 100% делоуд-недель выполнены | лог мезоциклов |
| Удержание | использование ≥ 12 недель подряд (2 полных мезоцикла) | история сессий |
| Сохранность данных | 0 потерь истории | наличие свежего бэкапа ≤ 2 нед |

## 5. Constraints

- **Один HTML-файл** без сборки и внешних зависимостей; полная работа офлайн (зал без сети).
- **Хранение — localStorage** браузера; обязателен экспорт/импорт JSON-бэкапа.
- **Mobile-first**: тёмная тема, крупные элементы, ввод одной рукой.
- **Один пользователь, одно устройство** — без аккаунтов, синхронизации и облака (v1).
- Методология зафиксирована: гибридная прогрессия (двойная прогрессия + RIR), календарные мезоциклы 5+1; альтернативные модели периодизации вне scope v1.
- Библиотека ~50 упражнений встроена в код; медиа-контент (видео техники) вне scope.
- **Roadmap v2 (вне scope MVP, но заложено в архитектуру):** видеоанализ техники — pose estimation в браузере (MediaPipe/TF.js), объективные метрики подхода (темп, амплитуда, симметрия) и rule-based проверки техники с рекомендациями («глубина падает к последним повторам → снизить вес»). MVP резервирует под это модель данных (поле `analysis` у подхода) и интерфейс сигналов движка.

## 6. Risks

| Риск | Вероятность | Impact | Митигация |
|---|---|---|---|
| Потеря данных localStorage (чистка кэша, обновление браузера) | Средняя | Критический | Настойчивое напоминание о бэкапе каждые 2 недели; экспорт в 1 клик |
| Систематическое завышение RIR пользователем → алгоритм гонит вес вверх | Высокая | Средний | Двойная прогрессия как база (объективные повторы), RIR — только корректор; консервативные пороги |
| Трение ввода выше ожидаемого → пользователь бросает логирование | Средняя | Критический | Прототип экрана тренировки тестируется первым; степперы, автоподстановка прошлых значений |
| Календарный делоуд не совпадает с реальной усталостью (бег, работа, сон) | Средняя | Средний | Ручной сдвиг делоуда ±1 неделя; в v2 — сигналы усталости |
| Стагнация из-за факторов вне приложения (сон, питание, стресс) — ложные подсказки детектора | Высокая | Низкий | Подсказки формулируются как гипотезы, не директивы |
| Проекция веса по Эпли неточна на дальних диапазонах (15+ повторов → цель 8) | Средняя | Низкий | Разведочный подход рекомендуется в 10–15 повторов; контрольный подход обязателен перед фиксацией веса |
| Переусложнение v1 (аналитика, конструктор) задержит ядро | Средняя | Средний | Приоритет сборки: экран тренировки → движок → аналитика → конструктор |
